üéØ Why Do We Learn Data Structures?

Data structures are the foundation of efficient programming. The way you organize your data directly impacts how fast your program runs and how much memory it uses. Choosing the right data structure can mean the difference between a program that runs in milliseconds versus one that takes hours. In interviews, competitive programming, and real-world software development, understanding data structures is essential for writing optimized, scalable code.

üì¶ Understanding Primitive Structures

Primitive data structures are the most basic building blocks provided directly by the programming language. They store single, simple values and are handled directly by the CPU.

The Four Primitives:

Integer: Whole numbers (e.g., -5, 0, 42). Used for counting, indexing, and arithmetic without decimals.

Real (Float/Double): Decimal numbers (e.g., 3.14, -0.001). Used for measurements, scientific calculations, and precise values.

Character: Single letters or symbols (e.g., 'A', '9', '@'). The building blocks of text and strings.

Boolean: True or False values only. Used for decisions, conditions, and flags in your programs.

üí° Why it matters: Everything in computing ultimately breaks down to these primitives. Understanding them helps you grasp memory allocation, type conversion, and why certain operations are faster than others.

üß© What Are Abstract Data Types (ADTs)?

Abstract Data Types are data structures defined by their behavior (what operations they support) rather than their implementation (how they're built internally). This is a powerful concept because it separates "what" from "how".

Example: A Stack ADT is defined by its operations: push (add to top), pop (remove from top), and peek (view top). Whether you implement it using an array or a linked list doesn't change what a Stack is ‚Äî it still follows Last-In-First-Out (LIFO) behavior.

üí° Why it matters: ADTs allow you to think at a higher level. You can design algorithms using Stacks, Queues, or Trees without worrying about implementation details until later. This is the foundation of good software architecture.

üìè Linear Data Structures Explained

Linear data structures arrange elements in a sequential, one-after-another order. Each element has a unique predecessor and successor (except the first and last).

üìö Array

A collection of elements stored in contiguous memory locations. Access any element instantly using its index (O(1) access time). Best when you know the size in advance and need fast random access.

Real-world example: A list of student grades where you access by roll number.

üì• Stack (LIFO - Last In, First Out)

Elements are added and removed from the same end (the "top"). Like a stack of plates ‚Äî you can only take from or add to the top.

Real-world examples: Undo/Redo in text editors, browser back button, function call stack.

üì§ Queue (FIFO - First In, First Out)

Elements are added at the rear and removed from the front. Like a line at a ticket counter ‚Äî first person in line gets served first.

Real-world examples: Print job scheduling, CPU task scheduling, message queues.

üîó Linked List

Elements (nodes) are scattered in memory but connected via pointers. Each node holds data and a reference to the next node. Great for frequent insertions/deletions.

Real-world examples: Music playlist (next/previous song), image viewer, blockchain.

üå≥ Non-Linear Data Structures Explained

Non-linear data structures don't arrange elements sequentially. Instead, elements can connect to multiple other elements, forming hierarchical or networked relationships.

üå≤ Tree

A hierarchical structure with a root node at the top and child nodes branching downward. Each node (except root) has exactly one parent. No cycles allowed.

Real-world examples: File system directories, organization charts, HTML DOM, family trees.

Special types: Binary Tree, Binary Search Tree (BST), AVL Tree, Heap, B-Tree.

üï∏Ô∏è Graph

A collection of vertices (nodes) connected by edges. Unlike trees, graphs can have cycles and nodes can have multiple connections. Most flexible structure.

Real-world examples: Social networks (friends), Google Maps (routes), internet (web pages), flight connections.

Types: Directed/Undirected, Weighted/Unweighted, Cyclic/Acyclic.

üéØ Set

A collection of unique elements with no duplicates and no specific order. Supports operations like union, intersection, and membership testing.

Real-world examples: Unique visitors to a website, available usernames, tags on a blog post.

Every time your program calls a function, the return address and local variables are pushed onto the call stack. This happens automatically behind the scenes! When a function returns, its data is popped off the stack.

Nonlinear Data Structures

üéØ Why Learn Nonlinear Data Structures?

The real world isn't always a straight line! Many relationships are hierarchical (like org charts) or networked (like social connections). Nonlinear structures model these naturally:

  • Trees: File systems, HTML DOM, organization charts, decision trees in AI
  • Graphs: Social networks (Facebook friends), Maps (Google Maps routes), Internet routing
  • Multidimensional Arrays: Images (2D pixel grids), spreadsheets, game boards

Many of the most important algorithms in computer science operate on trees and graphs!

In nonlinear data structures, data elements are not organized in a sequential fashion.
A data item in a nonlinear data structure could be attached to several other data elements to reflect a special relationship among them.
All the data items cannot be traversed in a single run.

Tree Data Structure

üéØ Why Learn Trees?

Trees are one of the most important data structures in computer science. They appear everywhere:

  • File Systems: Your computer organizes files in folders (directories) ‚Äî that's a tree!
  • HTML/DOM: Every web page is a tree of HTML elements
  • Databases: B-Trees power fast database lookups (MySQL, PostgreSQL)
  • AI/ML: Decision Trees for classification, Parse Trees for language processing
  • Compilers: Abstract Syntax Trees (AST) represent your code structure

If you understand trees, you understand how most software systems organize hierarchical data!

üí° Definition

A Tree is a nonlinear data structure that models a hierarchical organization.

The characteristic features are:

Each element may have several successors (called its "children")
Every element except one (called the "root") has a unique predecessor (called its "parent")

Trees are common in computer science:

  • Computer file systems are trees
  • The inheritance structure for C++/Java classes is a tree
What is a Tree?

In Computer Science, a tree is an abstract model of a hierarchical structure.

A tree consists of nodes with a parent-child relation.

üìò Applications
  • Organization Charts
  • File Systems
  • Programming Environment
Tree Definition
üìê Recursive Definition

Here is the recursive definition of an (unordered) tree:

A tree is a pair (r, S), where r is a node and S is a set of disjoint trees, none of which contains r.

  • The node r is called the root of the tree T
  • The elements of the set S are called its subtrees
  • The set S, of course, may be empty
  • The elements of a tree are called its nodes

Root, Parent and Children

If T = (x, S) is a tree, then:

  • x is the root of T and
  • S is its set of subtrees: S = {T‚ÇÅ, T‚ÇÇ, T‚ÇÉ, ..., T‚Çô}

Each subtree T‚±º is itself a tree with its own root r‚±º.

In this case, we call the node r the parent of each node r‚±º, and we call the r‚±º the children of r.

Basic Terminology
A node with no children is called a leaf.
A node with at least one child is called an internal node.
The Node having further sub-branches is called parent node.
Every node c other than the root is connected by an edge to some one other node p called the parent of c.
We also call c a child of p.

Connected Tree
üí° Connected Property

A tree is connected in the sense that if we start at any node n other than the root, move to the parent of n, to the parent of the parent of n, and so on, we eventually reach the root of the tree.

üìò Example

For instance, starting at n7, we move to its parent n4, and from there to n4's parent, which is the root n1.

Ancestors and Descendants
The parent-child relationship can be extended naturally to ancestors and descendants.
Informally, the ancestors of a node are found by following the unique path from the node to its parent, to its parent's parent, and so on.
The descendant relationship is the inverse of the ancestor relationship, just as the parent and child relationships are inverses of each other.

Height and Depth

üéØ Why Learn Height and Depth?

Height and depth are crucial metrics for analyzing tree performance:

  • Height determines efficiency: In a Binary Search Tree, search time is O(height). A balanced tree with height log(n) is MUCH faster than an unbalanced one with height n!
  • Depth affects memory: Recursive algorithms use stack space proportional to depth
  • Balancing algorithms: AVL trees and Red-Black trees maintain low height by tracking these values

Memory trick: Depth = how deep from top (root), Height = how high from bottom (leaves).

üìè Height of a Node n

The length of a longest path from n to a leaf.

Height of the tree = height of the root.

üìê Depth (Level) of a Node n

The length of the path from the root to n.

üéØ Why Learn the Tree ADT?

An Abstract Data Type (ADT) defines WHAT operations are available, not HOW they're implemented. The Tree ADT is like a "contract" that any tree implementation must follow. Understanding it helps you:

Think of ADT as the "interface" in object-oriented programming!

The tree ADT stores elements at positions, which are defined relative to neighboring positions.
Positions in a tree are its nodes, and the neighboring positions satisfy the parent-child relationships that define a valid tree.
Tree nodes may store arbitrary objects.

üéØ Why Learn Algorithm Complexity Analysis?

  • Identify bottlenecks ‚Äî Find the part of your code that's slowing everything down
  • Pass technical interviews ‚Äî "What's the time complexity?" is a standard interview question

We count primitive operations (comparisons, assignments, array accesses) as a function of input size n.

Analysis of the While Loop

Consider a random piece of code below where While loop is used:

Pseudocode - While Loop Analysis 
// COUNTING INSTRUCTIONS IN A WHILE LOOP
// Goal: Count total operations as a function of input size n

i = 0;                  // 1 time (initialization - OVERHEAD)

while (i < n) {         // n+1 times (checked n times TRUE + 1 time FALSE)
    stmt;               // n times (body executes when condition is true)
    i++;                // n times (increment happens n times)
}

// TOTAL INSTRUCTIONS: 1 + (n+1) + n + n = 3n + 2
// But simplifying: body operations = 2n, so f(n) = 2n + 1
üìê Solution

$$f(n) = 2n + 1$$

Breakdown: The "+1" is the initialization (overhead). The "2n" comes from stmt + i++ executing n times each.

Analysis of the For Loop

Consider another random piece of code below where for loop is used instead:

Pseudocode - For Loop Analysis 
// COUNTING INSTRUCTIONS IN A FOR LOOP
// for(init; condition; update) { body }

// i = 0     : runs 1 time (initialization)
// i < n     : runs n+1 times (condition check)
// i++       : runs n times (update after each iteration)
for (i = 0; i < n; i++) {
    stmt;               // n times (body statement)
    i++;                // n times (additional increment in body)
}

// TOTAL: 1 + (n+1) + n + n + n = 4n + 2
// Note: This example has i++ TWICE (loop + body) - unusual!
üìê Analysis

$$f(n) = 3n + 2$$ (with extra i++ in body)

$$f(n) = 2n + 1$$ (typical loop without extra increment)

Growth rate: $O(n)$ ‚Äî Both are LINEAR time complexity

üéØ Key Insight

Despite different exact formulas (2n+1 vs 3n+2), both have the same growth rate O(n). In Big-O analysis, we focus on how the function grows as n increases, not the exact constants.

Applying Theory of Algorithm Analysis

When applying the theory of algorithm analysis, one must sometimes be aware of the time it takes to execute:

The basic operation
The overhead instructions
The control instructions

on the actual computer on which the algorithm is implemented.

üîß Overhead Instructions

Instructions such as initialization instructions before a loop.

The number of times these instructions execute does not increase with input size.

üéÆ Control Instructions

Instructions such as incrementing an index to control a loop.

The number of times these instructions execute increases with input size.

üéØ Key Insight

The basic operation, overhead instructions, and control instructions are all properties of an algorithm and the implementation of the algorithm. They are NOT properties of a problem.

This means that they are usually different for two different algorithms for the same problem.

Array Lookup Example

Consider our Array Look up example:

The maximum element in an array can be looked up using a simple piece of code such as this piece of JavaScript code. Given an input array A of size n:

JavaScript - Finding Maximum Element 
// FIND MAXIMUM ELEMENT IN AN ARRAY
// Purpose: Find the largest value in array A of size n

// Step 1: Initialize M with the first element
// We assume the first element is the maximum until proven otherwise
var M = A[0];  // This is an OVERHEAD instruction (runs once)

// Step 2: Loop through all elements to find the maximum
for (var i = 0; i < n; ++i) {  // CONTROL instruction (runs n+1 times)
    
    // Step 3: Compare current element with maximum
    // This comparison ALWAYS happens - that's 1 instruction
    // The array lookup A[i] ALWAYS happens - that's another instruction
    if (A[i] >= M) {  // 2 instructions that ALWAYS execute
        
        // Step 4: Update maximum if current element is larger
        // This assignment MAY or MAY NOT run depending on values
        // When it runs: 1 array lookup + 1 assignment = 2 more instructions
        M = A[i];  // Conditional instructions (depends on input)
    }
}
// After loop: M contains the maximum element in array A
üí° Instruction Count Analysis

Looking at the for body, we have an array lookup operation and a comparison that happen always: that's two instructions there.

But the if body may run or may not run, depending on what the array values actually are.

If it happens to be so that A[i] >= M, then we'll run these two additional instructions:

  1. an array lookup
  2. and an assignment: M = A[i]

Java - Fully Commented 
// INSERTION SORT ALGORITHM
// Time Complexity: Best O(n), Average O(n¬≤), Worst O(n¬≤)
// Space Complexity: O(1) - sorts in place

// Outer loop: Start from index 1 (element at index 0 is already "sorted")
// Each iteration, we take element a[i] and insert it into sorted portion a[0:i-1]
for (int i = 1; i < a.length; i++) {
    
    // Step 1: Save the current element to be inserted
    // We need to save it because we'll overwrite this position during shifting
    int t = a[i];
    
    // Step 2: Declare j outside the loop so we can use it after the loop
    int j;
    
    // Step 3: Inner loop - shift elements right to make room for t
    // Condition 1: j >= 0 means we haven't gone past the beginning
    // Condition 2: t < a[j] means current element is larger than t
    // Both conditions must be true to continue shifting
    for (j = i - 1; j >= 0 && t < a[j]; j--) {
        // Shift element at j one position to the right
        // This creates a "gap" for inserting t
        a[j + 1] = a[j];
    }
    
    // Step 4: Insert t into its correct position
    // After the loop, j is at the position just before where t should go
    // So we insert at j + 1
    a[j + 1] = t;
}
üéØ Understanding Insertion Sort

Think of it like sorting playing cards: You pick up cards one by one, and insert each card into its correct position among the cards you've already sorted in your hand.

  • Best Case O(n): Array already sorted - inner loop never executes
  • Worst Case O(n¬≤): Array in reverse order - maximum shifts needed
  • Advantage: Very efficient for small arrays or nearly sorted data

Traversal Algorithms

üéØ Why Learn Traversal Algorithms?

A data structure is useless if you can't access all its elements! Traversal is the systematic way to visit every node exactly once. Different traversal orders serve different purposes:

  • Level Order: Print a tree level-by-level, find shortest path (BFS in graphs)
  • Preorder: Copy a tree, serialize structure, create prefix expressions
  • Postorder: Delete a tree safely (children before parent), calculate folder sizes
  • Inorder: Get sorted output from a Binary Search Tree (only for binary trees)

The traversal you choose depends on what problem you're solving!

üí° Definition

A traversal (circumnavigation) algorithm is a method for processing a data structure that applies a given operation to each element of the structure.

For example, if the operation is to print the contents of the element, then the traversal would print every element in the structure.
The process of applying the operation to an element is called visiting the element.
So executing the traversal algorithm causes each element in the structure to be visited.
Level Order Traversal
The level order traversal algorithm visits the root, then visits each element on the first level, then visits each element on the second level, and so forth.
Each time visiting all the elements on one level before going down to the next level.
If the tree is drawn in the usual manner with its root at the top and leaves near the bottom, then the level order pattern is the same left-to-right top-to-bottom pattern that you follow to read English text.

Binary Trees

üéØ Why Learn Binary Trees?

Binary trees are the most studied and most useful type of tree. They're simpler than general trees (max 2 children) but incredibly powerful:

  • Binary Search Trees (BST): Enable O(log n) search ‚Äî finding 1 item in 1 billion takes only ~30 comparisons!
  • Heaps: Power priority queues used in task scheduling, Dijkstra's algorithm
  • Expression Trees: Represent mathematical expressions like (3 + 5) √ó 2
  • Huffman Trees: Data compression (used in ZIP, JPEG, MP3)

The left/right distinction in binary trees enables efficient divide-and-conquer algorithms!

üìê Formal Definition

A binary tree is either the empty set or a triple T = (x, L, R), where x is a node and L and R are disjoint binary trees, neither of which contains x.

The node x is called the root of the tree T, and the subtrees L and R are called the left subtree and the right subtree of T rooted at x.

Terminologies

The definitions of the terms size, path, length of a path, depth of a node, level, height, interior node, ancestor, descendant, and subtree are the same for binary trees as for general trees.

üéØ Why Learn Full and Complete Binary Trees?

Not all binary trees are created equal! Full and Complete trees have special properties that make them extremely useful:

  • Heaps: Complete binary trees are the foundation of heap data structures used in priority queues and heap sort
  • Efficient storage: Complete trees can be stored in arrays WITHOUT pointers (saving memory)
  • Predictable height: Full trees with n nodes have height exactly log‚ÇÇ(n+1) - 1

üå≤ Full Binary Tree

A binary tree is said to be full if:

  • All its leaves are at the same level
  • Every interior node has two children

üå¥ Complete Binary Tree

A complete binary tree is either:

  • A full binary tree, OR
  • Full except for a segment of missing leaves on the right side of the bottom level

Why Complete Binary Trees Matter
üéØ Make a Note

Complete binary trees are important because they have a simple and natural implementation using ordinary arrays.

The natural mapping is actually defined for any binary tree: Assign the number 1 to the root; for any node, if i is its number, then assign 2i to its left child and 2i+1 to its right child (if they exist).

This assigns a unique positive integer to each node. Then simply store the element at node i in a[i], where a[] is an array.

Binary Tree Representations

üéØ Why Learn Binary Tree Implementations?

There are two main ways to store a binary tree in memory, and choosing the right one can make or break your application's performance:

  • Array Implementation: Best for complete/full binary trees (like heaps). No pointer overhead, cache-friendly, but wastes space for sparse trees.
  • Linked Implementation: Best for general binary trees. Flexible, no wasted space, but pointer overhead and cache-unfriendly.

Binary Tree Traversals

üéØ Why Learn Tree Traversals?

A tree stores data, but to use that data, you need to visit every node. The ORDER you visit nodes matters for different tasks:

  • Preorder (Root‚ÜíLeft‚ÜíRight): Copy a tree, create a prefix expression, serialize tree structure
  • Inorder (Left‚ÜíRoot‚ÜíRight): Get sorted order from a Binary Search Tree ‚Äî this is HUGE!
  • Postorder (Left‚ÜíRight‚ÜíRoot): Delete a tree safely, calculate folder sizes, evaluate expressions
  • Level Order (Breadth-First): Print tree level by level, find shortest path, serialize for storage

These four traversals are fundamental patterns you'll see in countless algorithms!

The three traversal algorithms that are used for general trees apply to binary trees as well: the preorder traversal, the postorder traversal, and the level order traversal.

1. Big-O Notation Explained Simply

When we analyze algorithms, we use Big-O notation to describe how the running time grows as the input size increases.

O(1) Constant Array access: a[5] Same time regardless of input size

üéØ Why Big-O Matters

For small inputs (n=10), any algorithm is fast. But for large inputs (n=1,000,000):

  • O(n) takes ~1 million operations
  • O(n¬≤) takes ~1 trillion operations (1000x slower!)
2. Recursion in Tree Traversals

Tree traversals use recursion - a function that calls itself. Here's how it works conceptually:

Pseudocode - Understanding Recursion 
// PREORDER TRAVERSAL (Root ‚Üí Left ‚Üí Right)
function preorder(node):
    if node is NULL:
        return              // Base case: stop if no node
    
    visit(node)             // 1. Process the current node FIRST
    preorder(node.left)     // 2. Then recursively process left subtree
    preorder(node.right)    // 3. Then recursively process right subtree

// INORDER TRAVERSAL (Left ‚Üí Root ‚Üí Right)
function inorder(node):
    if node is NULL:
        return
    
    inorder(node.left)      // 1. First process left subtree
    visit(node)             // 2. Then process the current node
    inorder(node.right)     // 3. Then process right subtree

// POSTORDER TRAVERSAL (Left ‚Üí Right ‚Üí Root)
function postorder(node):
    if node is NULL:
        return
    
    postorder(node.left)    // 1. First process left subtree
    postorder(node.right)   // 2. Then process right subtree
    visit(node)             // 3. Process the current node LAST
üìò Memory Aid

The name tells you when to visit the ROOT:

  • PREorder = visit root PRE (before) subtrees
  • INorder = visit root IN between subtrees
  • POSTorder = visit root POST (after) subtrees

Question 13: Five Binary Trees with Three Nodes
‚ùì Question

How many structurally different binary trees can be formed with exactly 3 nodes? List all of them.

‚úÖ Answer

There are 5 different binary trees with 3 nodes:

  1. Both Children: Root has left child AND right child
  2. Left-Left Chain: Root ‚Üí left child ‚Üí left grandchild
  3. Left-Right Zigzag: Root ‚Üí left child ‚Üí right grandchild
  4. Right-Left Zigzag: Root ‚Üí right child ‚Üí left grandchild
  5. Right-Right Chain: Root ‚Üí right child ‚Üí right grandchild

Why 5? Because left and right positions are DIFFERENT in binary trees. A left-left chain is NOT the same as a right-right chain.

Question 14: Full vs Complete Binary Tree
‚ùì Question

‚úÖ Answer

Complete Binary Tree: Either a full binary tree OR a tree that is full except for a segment of MISSING LEAVES ON THE RIGHT SIDE of the bottom level.

Key Difference: A full tree is "perfect" at all levels. A complete tree can have missing nodes, but ONLY on the right side of the last level.

Array Implementation:

Explain the "flag method" for determining traversal order. Where is the flag placed for each traversal type?

‚úÖ Answer

The flag method: Imagine a flag attached to each node. Walk around the tree starting from root, going left. When you pass a flag, you visit that node.

  • Preorder: Flag on the LEFT side of each node
  • Inorder: Flag on the BOTTOM of each node
  • Postorder: Flag on the RIGHT side of each node
